# Review - 审查与复盘

对指定代码或描述做**前端/体验向**审查，并可基于 `.cursor/plan/` 下规划文档进行**阶段性复盘**，核对任务完成度与里程碑。

## 使用方法

```bash
/review [代码、描述或规划文档路径]
```

- **有参数**：审查指定的代码片段、文件或需求描述（侧重前端/体验）
- **指向规划文档**：对该规划做阶段性复盘（任务清单、里程碑、Demo、偏差）

---

## Cursor 工具与工作目录

**工作目录**：
- 默认使用当前工作区根路径
- 若有多个工作区，先用 Glob/Grep 确定任务相关目录，或使用 **AskQuestion** 让用户选择目标工作区

**检索**：需要时使用 **SemanticSearch** 获取相关上下文；不足时用 **Grep**、**Glob** 补充。

---

## 执行工作流

### A. 代码/描述审查（前端与体验）

当用户提供的是**代码或描述**时：

#### 阶段 1：获取待审查内容与上下文

`[模式：研究]`

- 以用户指定的代码、文件或描述为审查对象
- 使用 **SemanticSearch**（及必要时 Grep/Glob）获取相关模块、设计规范、组件约定

#### 阶段 2：前端/体验向审查

`[模式：审查]`

由 **Cursor** 完成，聚焦**前端与体验**（不涉及后端逻辑审查）：
- **可访问性**：语义、ARIA、键盘与屏幕阅读器
- **响应式**：断点、布局、触摸与视口
- **设计一致性**：组件用法、样式系统、视觉层级
- **交互与状态**：反馈、加载、错误与边界状态

产出：按 **Critical / Major / Minor / Suggestion** 分类列出问题，每项含位置与修改建议。

#### 阶段 3：呈现审查结果

`[模式：总结]`

```markdown
## 📋 审查报告（前端/体验）

### 审查范围
<说明审查对象与涉及文件>

### 关键问题 (Critical) / 主要问题 (Major) / 次要问题 (Minor) / 建议 (Suggestions)
...

### 总体评价
- 体验与一致性：[优秀/良好/需改进]
```

---

### B. 基于规划的阶段性复盘

当用户**指定规划文档**（如 `.cursor/plan/xxx.md`）或明确要求「复盘」时：

#### 步骤 1：任务清单核对

- 打开对应规划文档，逐条检查任务项 `- [ ]` 是否应改为 `- [x]`（已完成则勾选）
- 避免 context 过长导致遗漏；可分段核对并更新同一文件

#### 步骤 2：里程碑更新

- 同步更新文档中的「〇、进度里程碑」表（若有）：
  - 各阶段完成度：已完成数 / 总任务数
  - **当前进度**：下一批待办简述
  - **整体完成度**：总已完成 / 总任务

#### 步骤 3：Demo 验收

- 各阶段 Demo 路由是否可访问、功能是否符合预期
- 将结果反映在里程碑或「验收说明」中

#### 步骤 4：偏差说明

- 若有任务取消、拆分或新增，在规划文档中注明（如「实现调整」小节），保持规划与实际执行一致

复盘完成后，可简要汇报：已完成数/总任务数、当前阶段、下一步建议。

---

## 关键规则

1. **不做 git 审查** – 不依赖 `git diff` / `git status`，仅针对用户给出的代码、描述或规划文档
2. **不覆盖后端审查** – 审查维度仅限前端、体验与一致性；后端问题不在此命令范围
3. **执行归属** – 审查与复盘均由 Cursor 在本会话完成；复盘时只更新规划文档中的清单与里程碑，不修改产品代码（除非用户明确要求顺带改代码）
