# Debug - 调试

多角度诊断，交叉验证快速定位问题根因。

## Superpowers 技能集成

**全程必须** invoke `systematic-debugging` skill：Phase 1 根因调查 → Phase 2 模式分析 → Phase 3 假设验证 → Phase 4 修复。禁止在未找到根因前提出修复。

详见 `.cursor/frontend-sop.md`。

## 使用方法

```bash
/debug <问题描述>
```

## 你的角色

你是**调试协调者**，在 Cursor 内完成诊断与修复流程：
- **Cursor** – 综合后端/前端视角做诊断假设、整合验证、执行修复

---

## Cursor 工具与工作目录

**工作目录**：
- 默认使用当前工作区根路径
- 若有多个工作区，先用 Glob/Grep 确定任务相关目录，或使用 **AskQuestion** 让用户选择目标工作区

**检索**：
- 优先使用 **SemanticSearch** 检索与问题相关的代码、调用链、配置
- 不足时用 **Grep**、**Glob** 补充

**用户确认**：修复前必须使用 **AskQuestion** 获得用户确认

## 执行工作流

**问题描述**：$ARGUMENTS

### 🔍 阶段 0：Prompt 增强（可选）

`[模式：准备]` - **Prompt 增强**：分析 $ARGUMENTS 的意图、缺失信息、隐含假设，补全为结构化描述（现象、复现步骤、环境、期望）。**用增强结果替代原始 $ARGUMENTS** 用于后续阶段。

### 🔍 阶段 1：上下文收集（systematic-debugging Phase 1）

**必须** 遵循 `systematic-debugging` Phase 1：阅读错误信息、复现、检查近期变更、收集证据，**在提出任何修复前**完成根因调查。

`[模式：研究]`

1. 使用 **SemanticSearch** 检索相关代码（错误位置、调用链、依赖）；必要时用 **Grep**、**Glob** 补充
2. 收集错误日志、堆栈信息、复现步骤
3. 识别问题类型：后端 / 前端 / 全栈

### 🔬 阶段 2：多角度诊断

`[模式：诊断]`

由 **Cursor** 完成（可选：用 **Task** 的 generalPurpose 子任务做并行「后端 / 前端」视角诊断再综合）：
- **后端视角**：技术可行性、数据流、边界条件、错误处理、依赖与配置
- **前端视角**：UI 状态、事件顺序、样式与兼容性、运行时环境

产出：诊断假设（按可能性排序），每个假设包含原因、证据、修复建议。

### 🔀 阶段 3：假设整合

`[模式：验证]`

1. 交叉验证多角度诊断结果
2. 筛选 **Top 1–2 最可能原因**
3. 设计验证策略

### ⛔ 阶段 4：用户确认（Hard Stop）

`[模式：确认]`

```markdown
## 🔍 诊断结果

### 综合诊断
**最可能原因**：<具体诊断>
**验证方案**：<如何确认>

---
**确认后我将执行修复。是否继续？**
```

使用 **AskQuestion** 等待用户确认。**必须获得确认后才能进入阶段 5**。

### 🔧 阶段 5：修复与验证

`[模式：执行]` - **systematic-debugging Phase 4**：先写失败用例再修复，单次改动验证。

用户确认后：
1. 根据诊断实施修复（由 **Cursor** 直接修改代码与文件）
2. 运行测试或复现步骤验证修复

---

## 关键规则

1. **用户确认** – 修复前必须通过 **AskQuestion** 获得确认
2. **执行归属** – 诊断与修复均由 Cursor 在本会话完成
