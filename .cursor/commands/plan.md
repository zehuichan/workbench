## 核心协议

- **语言协议**：与工具交互用**英语**，与用户交互用**中文**
- **止损机制**：当前阶段输出通过验证前，不进入下一阶段
- **仅规划**：本命令允许读取上下文与写入 `.cursor/plan/*` 计划文件，但**禁止修改产品代码**
- **执行归属**：所有分析与计划由 **Cursor** 在本会话完成；后续执行由用户在本会话或新会话中按计划实施

---

## Cursor 工具与工作目录

**工作目录**：

- 默认使用当前工作区根路径
- 若有多个工作区，先用 Glob/Grep 确定任务相关目录，或使用 **AskQuestion** 让用户选择目标工作区

**检索**：

- 优先使用 **SemanticSearch**（语义查询），用自然语言构建查询（Where/What/How）
- **禁止基于假设回答**
- SemanticSearch 不可用时，使用 **Grep** + **Glob** 进行文件发现与关键符号定位

**用户确认**：需要用户选择工作区或确认时，使用 **AskQuestion**

## 执行工作流

**规划任务**：$ARGUMENTS

### 🔍 Phase 1：上下文全量检索

`[模式：研究]`

#### 1.1 Prompt 增强（必须首先执行）

**Prompt 增强**：分析 $ARGUMENTS 的意图、缺失信息、隐含假设，补全为结构化需求（明确目标、技术约束、范围边界、验收标准）。**用增强结果替代原始 $ARGUMENTS** 用于后续所有阶段。

#### 1.2 资源参考（若用户提供）

若用户给出**资源/文献地址**（如参考项目、文档、源码链接），**先查阅该资源**再继续规划；将可复用的思路或约束纳入后续分析与计划。

#### 1.3 上下文检索

使用 **SemanticSearch** 检索与需求相关的代码与结构：

- 基于增强后需求构建语义查询（Where/What/How）
- 若检索不足，用 **Grep**、**Glob** 补充文件与符号

#### 1.4 完整性检查

- 必须获取相关类、函数、变量的**完整定义与签名**
- 若上下文不足，触发**递归检索**
- 优先输出：入口文件 + 行号 + 关键符号名；必要时补充最小代码片段（仅用于消除歧义）

#### 1.5 需求对齐

- 若需求仍有模糊空间，**必须**向用户输出引导性问题列表（可用 **AskQuestion**）
- 直至需求边界清晰（无遗漏、无冗余）

### 💡 Phase 2：分析与计划合成

`[模式：分析]` - **必须** 遵循 `writing-plans` skill 的 bite-sized 任务粒度与文档结构。

#### 2.1 多角度分析

由 **Cursor** 完成（可选：用 **Task** 的 generalPurpose 子任务做并行「前端专家」视角分析再综合）：

- **技术/实现视角**：技术可行性、组件架构、性能考量、潜在风险
- **体验视角**：UI/UX 影响、用户体验、信息架构、交互、可访问性、视觉一致性
- **性能优化**：在设计需求实现时尽量考虑性能瓶颈，并探讨优化方案

产出：多角度解决方案 + 优劣势分析。

#### 2.2 交叉验证

整合分析结果，进行迭代优化：

1. **识别一致观点**（强信号）
2. **识别分歧点**（需权衡）
3. **逻辑推演**：消除方案中的逻辑漏洞

#### 2.3（可选）计划草案

可由 **Cursor** 先产出“计划草案”（仍**不允许**修改产品文件），再整理为最终计划：

- 重点：信息架构、组件结构、交互、可访问性、视觉一致性、响应式

#### 2.4 生成实施计划（最终版）

综合分析结果，生成 **Step-by-step 实施计划**。规划文档须包含：

**〇、进度里程碑**（放在文档前部）：

- 表格：阶段、名称、状态（`✅ 已完成` / `🔶 进行中` / `⏳ 未开始`）、完成度（如 5/8）、Demo 路由
- **当前进度**：简短描述下一个待办
- **整体完成度**：M/N 任务（约 X%）
- 每阶段建议有对应 Demo 路由，便于验收

**功能拆解**：

- 按**阶段**拆分，每阶段有明确目标与产出
- 任务项使用 `- [ ]` / `- [x]` 格式并带编号（如 4.1.6），便于进度追踪与逐步执行
- 尽可能详细，按新特性逐步开发

**计划正文结构示例**：

```markdown
## 📋 实施计划：<任务名称>

### 〇、进度里程碑

| 阶段 | 名称   | 状态      | 完成度 | Demo      |
| ---- | ------ | --------- | ------ | --------- |
| 0    | 脚手架 | ✅ 已完成 | 9/9    | `/xxx`    |
| 1    | xxx    | 🔶 进行中 | 5/8    | `/stage1` |
| 2    | xxx    | ⏳ 未开始 | 0/10   | —         |

**当前进度：** 简短描述下一个待办
**整体完成度：** M/N 任务（约 X%）

### 技术方案

<综合多角度分析的最优方案>

### 实施步骤

1. <步骤 1> - 预期产物
2. <步骤 2> - 预期产物
   ...
（或按阶段列出任务清单，每项为 `- [ ]` 编号 描述）

### 关键文件

| 文件                    | 操作 | 说明 |
| ----------------------- | ---- | ---- |
| path/to/file.ts:L10-L50 | 修改 | 描述 |

### 风险与缓解

| 风险 | 缓解措施 |
| ---- | -------- |
```

### ⛔ Phase 2 结束：计划交付（非执行）

**本命令的职责到此结束，必须执行以下动作**：

1. 向用户展示完整实施计划（含伪代码）
2. 将计划保存至 **`.cursor/plan/<功能名>.md`**（功能名从需求中提取，如 `theme-switcher`、`settings-page` 等）
3. **核对**：里程碑表中的完成度数字与任务清单中的 `- [ ]`/`- [x]` 数量一致
4. 以**加粗文本**输出提示（必须使用实际保存的文件路径）：

   ***

   **📋 计划已生成并保存至 `.cursor/plan/实际功能名.md`**

   **请审查上述计划，您可以：**
   - 🔧 **修改计划**：告诉我需要调整的部分，我会更新计划
   - ▶️ **执行计划**：在本会话继续说明「按计划实施」，或在新会话中打开该计划文件并按步骤实施

   ***

   **⚠️ 注意**：上面的 `实际功能名.md` 必须替换为你实际保存的文件名！

5. **立即终止当前回复**（Stop here. No more tool calls.）

**⚠️ 绝对禁止**：

- ❌ 问用户 "Y/N" 然后自动执行（执行由用户明确发起）
- ❌ 对产品代码进行任何写操作
- ❌ 在用户未明确要求修改时继续触发额外工具调用

## 计划保存

规划完成后，将计划保存至：

- **首次规划**：`.cursor/plan/<功能名>.md`
- **迭代版本**：`.cursor/plan/<功能名>-v2.md`、`.cursor/plan/<功能名>-v3.md`...

计划文件写入应在向用户展示计划前完成。

---

## 计划修改流程

如果用户要求修改计划：

1. 根据用户反馈调整计划内容
2. 更新 `.cursor/plan/<功能名>.md` 文件
3. 重新展示修改后的计划
4. 再次提示用户审查或执行

---

## 后续步骤

用户审查满意后，在本会话或新会话中**按计划实施**（逐项执行步骤、修改产品代码）。无需单独“执行命令”，直接按计划文件中的步骤进行开发即可。

---

## 关键规则

1. **仅规划不实施** – 本命令不执行任何产品代码变更
2. **不问 Y/N** – 只展示计划，让用户决定下一步
3. **多角度综合** – 分析时聚焦前端与体验视角，由 Cursor 综合成一份计划
4. **计划路径** – 统一使用 `.cursor/plan/<功能名>.md`
