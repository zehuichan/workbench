## 一、需求分析与增强

### 1.1 原始需求

列配置新增 `dependencies` 字段，允许添加字段之间的依赖项，以根据其他字段的值控制字段：

```ts
{
  prop: 'status',
  dependencies: {
    triggerFields: ['name'],
    disabled(values, api) {},
    trigger(values, api) {},
    rules(values, api) {},
    required(values, api) {},
    componentProps(values, api) {},
  },
}
```

### 1.2 增强后需求（结构化）

| 维度       | 描述                                                                                                            |
|----------|---------------------------------------------------------------------------------------------------------------|
| **目标**   | 在列配置上声明式描述字段间依赖关系，当 `triggerFields` 中任一字段值变动时，自动计算被依赖字段的动态状态并应用                                             |
| **范围**   | 仅限 `ReTableNext` 组件内部，不改变对外 `columns` 配置结构的兼容性                                                               |
| **API**  | `dependencies: ColumnDependencies<T>` — 含 `triggerFields`、`disabled`、`trigger`、`rules`、`required`、`componentProps` |
| **回调签名** | `(values: T, api: DependencyApi<T>) => ReturnType`，其中 `values` 为当前行数据，`api` 提供字段读写与上下文信息                        |
| **触发时机** | 编辑确认（confirmEdit）后，对变更字段查找所有依赖方并执行评估；`all` 模式下值直接写入 row，Cell 通过响应式自动重新评估                                       |
| **验收**   | Demo 页面演示：字段联动禁用、联动赋值、动态下拉选项、动态必填/校验规则、条件显隐                                                                  |

## 二、技术方案

### 2.1 核心类型设计

```ts
/** 依赖 API — 传给各回调函数的第二参数 */
export interface DependencyApi<T = RowData> {
  /** 当前行索引 */
  rowIndex: number;
  /** 当前行数据（引用） */
  row: T;
  /** 当前列配置 */
  column: ReTableNextColumn<T>;
  /** 读取当前行任意字段值 */
  getFieldValue: (prop: string) => any;
  /** 设置当前行任意字段值（直接写入 row，同步标记 dirty） */
  setFieldValue: (prop: string, value: any) => void;
}

/** 依赖配置 — 挂载在列配置的 dependencies 字段 */
export interface ColumnDependencies<T = RowData> {
  /** 监听哪些字段的值变动 */
  triggerFields: string[];
  /** 动态禁用（true → 阻止进入编辑 + 编辑器 disabled） */
  disabled?: (values: T, api: DependencyApi<T>) => boolean;
  /** 字段变更后的副作用（联动赋值等） */
  trigger?: (values: T, api: DependencyApi<T>) => void;
  /** 动态校验规则 */
  rules?: (values: T, api: DependencyApi<T>) => RuleItem | RuleItem[];
  /** 动态必填 */
  required?: (values: T, api: DependencyApi<T>) => boolean;
  /** 动态编辑组件 props */
  componentProps?: (values: T, api: DependencyApi<T>) => Record<string, any>;
}

/** 依赖解析结果 — Cell 组件消费 */
export interface DependencyState {
  disabled: boolean;     // disabled() 结果
  required?: boolean;    // required() 结果
  rules?: RuleItem | RuleItem[];  // rules() 结果
  componentProps?: Record<string, any>;  // componentProps() 结果
}
```

### 2.2 Composable 架构

```
use-dependencies.ts
├── buildTriggerMap()       # triggerField → [依赖列] 映射
├── createDependencyApi()   # 构造 DependencyApi 实例
├── resolveDependencyState()# 解析单个单元格的依赖状态
└── onFieldChange()         # 触发字段变更后的 trigger 回调
```

### 2.3 数据流

```
confirmEdit() → 值写入 row
  ↓
confirmEditWithHistory() → 遍历 changes
  ↓
onFieldChange(rowIndex, changedProp) → 查 triggerMap
  ↓
对每个依赖列执行 trigger(row, api)
  ↓（trigger 中可调用 api.setFieldValue）
  ↓
row[prop] 更新 → Vue 响应式触发 Cell 重新渲染
  ↓
Cell render 中 resolveDependencyState(rowIndex, column) 重新计算
  ↓
应用 disabled / componentProps / if / show / required 到 UI
```

### 2.4 关键设计决策

| 决策                      | 选择                                                              | 理由                                               |
|-------------------------|------------------------------------------------------------------|--------------------------------------------------|
| 依赖评估时机                  | Cell 渲染时响应式计算（disabled/required/componentProps）                | Vue 原生响应式，row 数据变更自动触发重新评估；无需额外 watch/cache       |
| trigger 执行时机             | confirmEdit 后集中执行                                                | 明确的变更边界，避免编辑过程中副作用干扰；all 模式下由 cell-value-change 驱动 |
| disabled 语义              | 阻止进入编辑 + 视觉灰化 + all 模式下 editor 传 disabled                      | 与 `editable: false` 语义一致但动态化                     |
| values 参数                | 当前行数据对象 `row`（已提交值）                                             | 简单直观；编辑中的暂存值通过 api.getFieldValue 可选获取             |
| 级联触发（trigger 中 set 的值再触发） | v1 不支持级联，仅一层深度                                                  | 避免循环依赖与无限递归，复杂场景可由业务在 trigger 内手动处理               |
| DependencyApi.setFieldValue | 直接写入 `row[prop]` + 标记 dirty                                     | 副作用立即生效，与响应式系统一致                                 |

### 2.5 与现有系统的集成点

| 现有模块                        | 集成方式                                                                        |
|-----------------------------|-----------------------------------------------------------------------------|
| **types.ts**                | `ReTableNextColumn` 新增 `dependencies?: ColumnDependencies<T>`               |
| **re-table-next.vue**       | 初始化 `useDependencies`，provide `resolveDependencyState` + 绑定 `onFieldChange` |
| **re-table-next-cell.vue**  | 消费 `depState`：`disabled` / `componentProps` 合并                               |
| **re-table-next-column.vue** | 动态 `required` 指示器 + `columnBindings` 过滤 `dependencies`                      |
| **use-editable.ts**         | `isCellEditable` 增加 `disabled` 判断分支                                         |
| **use-validation.ts**       | `mergeRules` 增加 `dependencies.rules` + `dependencies.required` 动态解析         |
| **composables/index.ts**    | 导出 `useDependencies`                                                        |

### 2.6 性能考量

| 场景           | 策略                                                  |
|--------------|-----------------------------------------------------|
| 大量行 × 多依赖    | Cell 内 computed 惰性求值，仅在 row 数据实际变更时重新计算              |
| triggerMap 重建 | columns 变更时重建（computed），列配置通常稳定                      |
| trigger 副作用   | 仅在 confirmEdit 后执行，不在渲染路径中                           |
| resolveDependencyState | 函数调用轻量（调用用户回调），不做缓存（Vue 的响应式已覆盖） |

### 2.7 风险与缓解

| 风险                      | 缓解措施                                                |
|-------------------------|-----------------------------------------------------|
| trigger 中 setFieldValue 导致循环 | v1 不支持级联；在 onFieldChange 内加递归保护（正在处理中则跳过）          |
| 用户回调抛异常                 | try-catch 包裹所有用户回调，console.warn 提示                   |
| all 模式下值直接写入 row        | if/show/disabled/componentProps 本身就是响应式计算，天然支持；trigger 通过 watch 或事件驱动 |
| dependencies 回调函数频繁调用     | 文档告知回调应为纯函数或轻量操作；componentProps 回调应避免创建大对象           |

### 5.2 use-dependencies.ts 核心逻辑

```ts
const DEFAULT_STATE: DependencyState = { disabled: false };

function buildTriggerMap(columns) {
  // triggerField → [{ column, dep }]
}

function createDependencyApi(rowIndex, row, column, onSetField?) {
  return {
    rowIndex, row, column,
    getFieldValue: (prop) => row[prop],
    setFieldValue: (prop, value) => {
      row[prop] = value;
      onSetField?.(rowIndex, prop);
    },
  };
}

function resolveDependencyState(rowIndex, column) {
  const dep = column.dependencies;
  if (!dep) return DEFAULT_STATE;
  const row = data.value[rowIndex];
  if (!row) return DEFAULT_STATE;
  const api = createDependencyApi(rowIndex, row, column);
  return {
    disabled: dep.disabled ? safecall(dep.disabled, row, api) : false,
    required: dep.required ? safecall(dep.required, row, api) : undefined,
    rules: dep.rules ? safecall(dep.rules, row, api) : undefined,
    componentProps: dep.componentProps ? safecall(dep.componentProps, row, api) : undefined,
  };
}

function onFieldChange(rowIndex, changedProp) {
  if (processing) return;  // 递归保护
  processing = true;
  try {
    const dependents = triggerMap.get(changedProp);
    for (const { column, dep } of dependents) {
      if (dep.trigger) safecall(dep.trigger, row, api);
    }
  } finally { processing = false; }
}
```
