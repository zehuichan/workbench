# ERP Document Linkage Demos Design

Date: 2026-07-17

> **Superseded** by [`2026-07-17-erp-emit-effect-demos-design.md`](./2026-07-17-erp-emit-effect-demos-design.md).
> Keep this file only as historical context for the shared-shell approach.

## Goal

在 Component Labs 中增加三个可交互的 ERP 单据场景，完整演示两类常见联动：

1. 明细变更后重算当前行并汇总到表头。
2. 表头变更后按明确策略传播到明细，再由明细反向汇总表头。

Demo 需要让传播策略、人工覆盖保护、计算顺序、服务端权威重算和并发冲突可观察，同时保持 PlusTable 为通用明细编辑组件，不把 ERP 业务规则加入其公共 API。

## Confirmed Decisions

- 使用销售订单、采购订单、费用报销三个业务场景。
- 每个场景使用独立路由，导航归入新分组“ERP 场景”。
- 使用配置驱动的共享场景壳；三个路由页面只选择场景配置。
- 不修改 PlusTable 公共 API。
- 提供内存模拟服务端，覆盖延迟保存、服务端重算、版本递增和并发冲突。
- 页面显式展示字段来源、受影响范围、联动流水和服务端结果。
- 沿用现有 `DemoPage`、`DemoBlock`、Element Plus 和 PlusTable 的文档站视觉体系。

## Architecture

建议文件结构：

```text
src/
  components/erp-document-demo/
    erp-document-demo.vue
    document-linkage.ts
    mock-document-server.ts
    types.ts
    scenarios/
      sales-order.ts
      purchase-order.ts
      expense-report.ts
      index.ts
  views/erp/
    sales-order-linkage-demo.vue
    purchase-order-linkage-demo.vue
    expense-report-linkage-demo.vue
```

职责边界：

- `erp-document-demo.vue`
  - 渲染表头表单、明细 PlusTable、汇总、工具栏、状态和联动流水。
  - 把用户操作转换为单据级命令。
  - 在需要覆盖值时先展示影响范围并请求确认。
  - 管理客户端修订号、未保存状态和模拟保存生命周期。
- `document-linkage.ts`
  - 提供无 UI 依赖的纯函数：传播计划、原子应用、行重算、表头汇总、校验和差异比较。
  - 固定执行顺序，避免表头 watcher 与明细 watcher 互相触发。
- `mock-document-server.ts`
  - 保存每个 Demo 实例的内存服务端快照。
  - 独立校验与重算传入快照，执行乐观锁检查，返回规范化结果。
- `scenarios/*.ts`
  - 定义表头字段、明细列、初始数据、传播策略、定价规则、行计算、汇总和业务校验。
- `views/erp/*.vue`
  - 薄路由页面，只把对应场景配置交给共享场景壳。

这些模块是 Demo 内部实现，不从 PlusTable 或仓库级公共入口导出。

## Core Data Model

单据草稿包含：

- `documentNo`：单据编号。
- `status`：本 Demo 固定为草稿状态。
- `clientRevision`：每次成功应用本地命令后递增。
- `serverVersion`：最后一次同步到客户端的服务端版本。
- `header`：场景定义的表头值。
- `lines`：场景定义的明细值。
- `summary`：由明细计算得到的只读汇总。
- `dirty`：本地是否存在未保存变化。

每条明细额外包含：

```ts
type FieldSource = 'inherited' | 'manual';

interface DocumentLine {
  id: string;
  fieldSources: Record<string, FieldSource>;
  [field: string]: unknown;
}
```

只有允许人工覆盖且会从表头传播的字段需要记录来源。计算字段不记录来源，也不可直接编辑。

场景配置提供：

- 页面标题、说明和规则说明。
- 表头字段描述和编辑器配置。
- PlusTable 明细列配置。
- 初始单据工厂。
- 表头字段变化后的传播计划函数。
- 明细行计算、整单汇总和业务校验函数。
- 模拟服务端规范化规则。

## Linkage Command Pipeline

所有联动通过显式命令执行，不用跨表头与明细的双向 watcher：

```text
接收用户命令
→ 计算传播计划
→ 必要时确认影响范围
→ 在草稿副本上原子应用
→ 重算受影响明细行
→ 从全部明细重算表头汇总
→ 执行客户端整单校验
→ 提交新草稿并递增 clientRevision
→ 写入联动流水
```

### Detail to Header

明细单元格提交后：

1. 若字段支持继承，将该字段来源标记为 `manual`。
2. 重算当前行的派生金额。
3. 从全部明细重新生成汇总，不对旧汇总做增量加减。
4. 记录“明细 → 表头”流水及变化值。

新增行继承当前表头默认值，继承字段来源初始化为 `inherited`。删除行后重新生成整单汇总。

### Header to Detail

每个表头字段明确声明以下一种策略：

- `force`：所有明细同步，不保留行级不同值。
- `inherit`：只更新来源为 `inherited` 的明细；来源为 `manual` 的明细保留并记录原因。
- `recalculate`：把表头值作为计算上下文，只重算仍由系统维护的行级价格或金额。

命令先生成包含“更新行、保留行、覆盖字段、重算字段”的传播计划。需要覆盖人工值或造成数据清空时，用户确认前不修改原草稿；任一步失败时整次命令不落地。

## Scenario Rules

### Sales Order

表头字段：

- 客户：触发系统维护价格重新定价。
- 价目表：触发系统维护价格重新定价。
- 币种：强制同步所有明细。
- 发货仓库：只传播到继承行。
- 默认税率：只传播到继承行。

明细字段：

- 商品、数量、币种、发货仓库、单价、折扣率、税率。
- 单价、仓库和税率记录来源。
- 未税金额、税额和含税金额为只读计算字段。

计算顺序：

```text
成交单价 = 单价 × (1 - 折扣率)
未税金额 = 数量 × 成交单价
税额 = 未税金额 × 税率
含税金额 = 未税金额 + 税额
```

表头汇总总数量、未税金额、税额和订单总额。

### Purchase Order

表头字段：

- 供应商：触发系统维护采购价重新定价。
- 采购组织：强制同步所有明细。
- 结算币种：强制同步所有明细。
- 收货仓库：只传播到继承行。
- 默认税率：只传播到继承行。

明细字段：

- 物料、数量、采购组织、币种、收货仓库、采购单价、税率。
- 采购单价、仓库和税率记录来源。
- 未税金额、进项税额和价税合计为只读计算字段。

表头汇总采购数量、未税金额、进项税额和采购总额。

### Expense Report

表头字段：

- 报销人部门：只传播到继承行。
- 默认项目：只传播到继承行。
- 报销币种：强制同步所有明细。
- 汇率：作为计算上下文重算全部本位币金额。

明细字段：

- 费用类型、发生日期、部门、项目、币种、原币金额、可抵扣税额。
- 部门和项目记录来源。
- 本位币金额为只读计算字段。

计算顺序：

```text
可报销原币金额 = max(原币金额 - 可抵扣税额, 0)
本位币金额 = 可报销原币金额 × 汇率
```

表头汇总原币申请金额、可抵扣税额和本位币报销金额。

## Mock Server

每个页面创建独立的内存服务端实例：

1. 客户端发送完整单据快照、`clientRevision` 和 `serverVersion`。
2. 服务端等待固定模拟延迟。
3. 服务端比较请求版本与当前内存版本；不一致时返回版本冲突。
4. 服务端忽略客户端汇总和计算金额，从原始表头与明细重新计算。
5. 服务端执行必填、数值范围和场景业务规则校验。
6. 成功后保存规范化快照，版本加一，并返回规范化结果和校正差异。

“模拟并发更新”操作只更新内存服务端版本，用于稳定复现冲突。冲突不会覆盖当前客户端草稿；页面提示当前客户端版本和服务端版本，并提供重新载入服务端快照的操作。

“重新载入服务端版本”会先确认是否放弃本地未保存修改，确认后以服务端快照替换草稿并清除 `dirty`。“重置”只恢复最近一次成功同步的服务端快照；尚未保存过时恢复场景初始快照。

服务端状态不写入 localStorage，刷新页面后恢复场景初始数据。

## Validation and Error Handling

- 表头必填错误显示在对应 Element Plus 表单项附近。
- 明细基础错误使用 PlusTable 的列规则和全表校验能力。
- 跨字段、跨行和整单业务错误显示在汇总错误区，并在可能时关联行号。
- 金额统一按两位小数规范化，避免浮点显示差异。
- 数量、单价、金额和汇率不得为负；必须大于零的字段由场景规则单独声明。
- 保存期间禁用表头、明细和单据操作，避免响应覆盖并发产生的新草稿；失败后恢复编辑并保持 `dirty`。
- 批量传播在副本中完成，异常时不提交部分更新。

## User Interface

每个页面包含：

1. 页面说明：解释该场景覆盖的联动策略。
2. 单据状态栏：单据编号、草稿状态、客户端修订号、服务端版本号和未保存状态。
3. 表头表单：字段旁显示 `强制同步`、`继承传播` 或 `触发重算` 标签。
4. 明细 PlusTable：支持编辑、新增、删除和字段来源展示。
5. 汇总区：显示该场景的关键金额和数量。
6. 操作区：保存、重置、模拟并发更新、重新载入服务端版本。
7. 联动流水：按时间倒序显示方向、命令、更新行、保留人工值和服务端校正。

批量传播确认框展示受影响行数、保留行数及将被覆盖的字段。页面在窄屏下改为单列布局，不改变现有全局视觉风格。

  - 校验失败不更新服务端快照。

  - 三个 ERP 路由及导航顺序。
- Demo 内容契约测试
  - 三个页面保留说明、规则、操作提示和共享场景壳引用。

## Acceptance Criteria

1. 导航出现“ERP 场景”分组及三个独立页面，现有路由行为不变。
2. 每个页面至少展示一次表头到明细和明细到表头联动。
3. `inherited` 字段会随表头更新，`manual` 字段按策略保留或经确认覆盖。
4. 所有联动严格遵循“传播 → 行重算 → 表头汇总 → 校验”的顺序且无循环触发。
5. 新增和删除明细后汇总立即正确更新。
6. 模拟保存由服务端重新计算并递增版本。
7. 模拟并发更新能够稳定产生版本冲突，且不会覆盖本地草稿。
8. 联动流水能说明更新、保留和服务端校正的原因。
9. 三个页面在常见桌面宽度和窄屏下均可操作。
10. `pnpm test`、`pnpm typecheck` 和 `pnpm build` 全部通过。

## Out of Scope

- 真实后端、数据库、消息队列或持久化草稿。
- 审批流、权限控制、库存占用和预算占用。
- 将 ERP 联动能力加入 PlusTable 公共 API。
- 抽取仓库级公开 composable 或 npm 包。
- 国际化和大数据量性能压测。
