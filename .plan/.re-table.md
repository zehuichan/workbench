## 一、原组件问题复盘

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| 1 | 605 行单文件，零 composable 拆分 | 高 | 导航、编辑、校验、脏追踪全部耦合在一个文件 |
| 2 | 热键 switch-case 硬编码 | 高 | 不可扩展、不可覆写，新增快捷键必须改源码 |
| 3 | `dirtyCells` 全量 O(n×m) 对比 | 中 | 每次响应式触发都重新遍历全部数据 × 全部列 |
| 4 | `cachedData` 无条件 deepClone | 中 | 即使不编辑也会 clone 全量数据 |
| 5 | `changeInstance` hack 暴露方法 | 中 | 通过覆写 `vm.exposed` 方式不安全，TS 推断丢失 |
| 6 | 全局事件手动绑定/解绑 | 低 | 未使用 `useEventListener`，KeepAlive 场景易泄漏 |
| 7 | 校验无视觉反馈 | 中 | `show-message: false`，用户看不到错误 |
| 8 | 仅注册 `Input` 一个编辑组件 | 中 | 无法选择日期、数字、下拉等 |
| 9 | 模板里遗留 `<pre>{{ dirtyCells }}</pre>` 调试代码 | 低 | 生产代码中不应出现 |

## 二、设计原则

1. **Composable-First**：每个独立关注点抽取为 composable，主组件仅做组装
2. **配置驱动 + 渲染函数兜底**：优先 `columns` JSON 配置，复杂场景通过 `render` / 插槽扩展
3. **渐进式编辑**：不编辑时零开销，编辑态按需挂载表单/组件
4. **类型安全**：所有 API 均有完整 TS 类型，支持泛型行数据
5. **向后兼容**：el-table 原生 props/events/slots 透传，不破坏已有用法

## 四、核心类型定义

> 以下为 `types.ts` 中的核心接口，开发阶段会持续迭代。

```ts
/** 泛型行数据 */
export type RowData = Record<string, any>

/** ──── 列配置 ──── */
export interface ReTableColumn<T = RowData> {
  // 基础（对齐 el-table-column）
  type?: 'selection' | 'index' | 'expand'
  prop?: keyof T & string
  label?: string
  width?: number | string
  minWidth?: number | string
  fixed?: 'left' | 'right' | boolean
  sortable?: boolean | 'custom'
  align?: 'left' | 'center' | 'right'

  // 显隐
  hidden?: boolean | ((column: ReTableColumn<T>) => boolean)

  // 编辑
  editable?: boolean | ((row: T) => boolean)
  component?: string | Component
  componentProps?: Record<string, any> | ((row: T, column: ReTableColumn<T>) => Record<string, any>)
  componentSlots?: Record<string, (...args: any[]) => VNode>

  // 校验
  required?: boolean                     // 表头必填星号
  rules?: RuleItem | RuleItem[]          // 列级校验规则（补充表级 rules）

  // 渲染
  formatter?: (row: T, column: ReTableColumn<T>, cellValue: any, rowIndex: number) => any
  renderHeader?: (column: ReTableColumn<T>) => VNode
  renderCell?: (scope: CellScope<T>) => VNode
  renderEditor?: (scope: CellScope<T>) => VNode

  // 交互
  disabled?: boolean | ((row: T) => boolean)
  readonly?: boolean | ((row: T) => boolean)
  tooltip?: string                       // 表头 tooltip
  cellClassName?: string | ((scope: CellScope<T>) => string)

  [key: string]: any                     // 透传 el-table-column 其余属性
}

/** 单元格作用域 */
export interface CellScope<T = RowData> {
  row: T
  column: ReTableColumn<T>
  rowIndex: number
  colIndex: number
}

/** ──── 表格 Props ──── */
export interface ReTableProps<T = RowData> {
  // 数据
  data?: T[]
  rowKey?: string | ((row: T) => string | number)
  columns?: ReTableColumn<T>[]
  loading?: boolean

  // 编辑
  editable?: boolean | 'row' | 'cell' | 'manual'

  // 校验
  rules?: Record<string, RuleItem | RuleItem[]>
  validateTrigger?: 'change' | 'blur' | 'manual'   // 默认 'manual'
  validateOnCellExit?: boolean                       // 默认 false

  // 热键
  hotkeys?: HotkeyBinding[]                         // 自定义 / 覆盖热键
  hotkeyEnabled?: boolean                            // 默认 true

  // 列管理
  tableKey?: string                                  // 用于列持久化存储
  columnSetting?: boolean                            // 列设置按钮
  columnSettingStorage?: 'local' | 'session' | false

  // 分页
  pagination?: false | PaginationConfig
}

/** ──── 热键 ──── */
export interface HotkeyBinding {
  key: string                           // 'ArrowUp' | 'Ctrl+C' | 'Ctrl+Shift+Z'
  handler: (ctx: HotkeyContext) => void | boolean
  when?: (ctx: HotkeyContext) => boolean
  preventDefault?: boolean              // 默认 true
  stopPropagation?: boolean             // 默认 false
  override?: boolean                    // true 时覆盖同名内置热键
}

export interface HotkeyContext {
  event: KeyboardEvent
  cellMeta: CellMeta
  row: RowData | null
  column: ReTableColumn | null
  tableData: RowData[]
  columns: ReTableColumn[]
  navigate: (rowDelta: number, colDelta: number) => void
  startEdit: () => void
  cancelEdit: () => void
  updateCellValue: (value: any) => void
}

/** ──── 编辑历史 ──── */
export interface EditRecord {
  rowIndex: number
  colIndex: number
  prop: string
  oldValue: any
  newValue: any
  timestamp: number
}

/** ──── 单元格元数据 ──── */
export interface CellMeta {
  row: number
  col: number
  activated: boolean
  editing: boolean       // 原 editable → 改名为 editing，语义更清晰
  contextmenu: boolean
}

/** ──── 分页 ──── */
export interface PaginationConfig {
  currentPage?: number
  pageSize?: number
  pageSizes?: number[]
  total?: number
  layout?: string
}
```

### 阶段 3：数据校验增强（Week 4）

> 目标：实时校验、可视化错误、联动校验

#### 3.1 `use-validation.ts` — 校验引擎

```ts
export function useValidation(options: {
  data: Ref<RowData[]>
  columns: Ref<ReTableColumn[]>
  tableRules: Ref<Record<string, RuleItem[]>>
  trigger: Ref<'change' | 'blur' | 'manual'>
  validateOnCellExit: Ref<boolean>
}) {
  const errors: Ref<Map<string, string>>  // key: `${rowIndex}-${prop}`

  /** 校验指定范围 */
  async function validate(rows?: boolean | number | number[]): Promise<ValidationResult>

  /** 校验单个字段 */
  async function validateField(rowIndex: number, prop: string): Promise<boolean>

  /** 清除校验错误 */
  function clearValidation(rowIndex?: number, prop?: string): void

  /** 滚动到第一个错误 */
  function scrollToFirstError(): void

  return { errors, validate, validateField, clearValidation, scrollToFirstError }
}
```

**校验规则合并策略：**

```
最终规则 = 表级 rules[prop] + 列级 column.rules
```

## 七、技术决策记录

### D1: 热键引擎自研 vs 第三方库

**决策：自研**

- 项目已有 `@vueuse/core`，`useEventListener` 辅助事件管理即可
- 核心需求是与表格编辑状态深度绑定（编辑态/非编辑态行为不同），第三方库的 scope 机制不匹配
- 控制在 ~200 行，维护成本可接受

### D2: 去掉 el-form 包裹，直接使用 async-validator

**决策：直接使用 async-validator**

- 减少每个可编辑单元格 2 层 DOM 嵌套
- 校验逻辑完全可控，不受 el-form 限制
- Element Plus 内部也是使用 async-validator，API 兼容
- 自定义错误展示（tooltip）比 el-form 的 show-message 更灵活

### D3: cellMeta 单对象 vs 拆分

**决策：拆分为 `activeRow` / `activeCol` / `isEditing`**

- 单对象 ref 任何字段变化都触发所有依赖更新
- 拆分后只有关心特定字段的组件才会重渲染
- 对于 200 行 × 10 列 = 2000 个单元格的场景，减少大量不必要的 re-render

### D4: 组件命名 ReTable

**决策：`ReTable`（CSS class: `re-table`）**

- 原组件 CSS class 已经是 `re-table`，保持一致
- `Re` 可理解为 "Reactive Enhanced" 或 "Reimagined"
- 与原 `PlusTable` 区分，避免迁移期冲突

### D5: 编辑历史组件级 vs 全局

**决策：组件级**

- 每个表格实例独立维护历史栈
- 组件卸载时自动清理，无内存泄漏风险
- 预留 `provide/inject` 注入点，未来可升级为跨表格方案

### D6: 列持久化存储 key

**决策：`re-table:${tableKey}:columns`**

- `tableKey` prop 必须由用户提供（无默认值时不持久化）
- 不再拼接 `route.path`，避免路由变化导致配置丢失
- 用户可通过 `tableKey` 完全控制存储粒度
